Q1

原始 IS 比较 · Raw IS Slippage Comparison

三种方法交叉验证：直接比较两算法的 IS 均值

3-Method Cross-Validation ▼

Goal: Test whether E[IS_algo1] ≠ E[IS_algo2] — using three methods to cross-validate.

Method 1: Welch t-test

Why NOT standard t-test? Standard t-test requires equal variance across groups. But algo_1 & algo_2 are routed to different order difficulties, so Var(IS) cannot be equal. Welch's test relaxes this assumption.
t_stat, p_val = stats.ttest_ind(g1, g2, equal_var=False) # Welch

Method 2: Mann-Whitney U

Even after Winsorization, IS distribution remains fat-tailed. Mann-Whitney tests:

P(IS_algo1 < IS_algo2) vs. P(IS_algo2 < IS_algo1)

→ Randomly picked algo_1 order: how likely is its IS smaller than algo_2's?

u_stat, u_p = stats.mannwhitneyu(g1, g2, alternative='two-sided')

Method 3: Bootstrap CI (N = 10,000)

  • •No prior assumption of distribution — simulates uncertainty by resampling
  • •Resample 10,000 times to compute mean of each algo's IS
  • •boots = IS_algo1_mean − IS_algo2_mean

What CI tells us:

  • ①(Direction) If CI = [a,b] and b < 0 → algo_1 IS < algo_2 IS
  • ②(Variance) Width of CI = measure of variance/uncertainty
boots = [np.mean(np.random.choice(g1,n,True)) - np.mean(np.random.choice(g2,n,True)) for _ in range(10_000)]

Q1 Conclusion

algo_1's IS is significantly smaller than algo_2's IS. BUT — this difference cannot be attributed to the algorithm. It is probably routing bias (Q3 will verify). Must continue to causal regression.

Q2

名义价值加权 IS · Notional-Weighted IS + Pareto

简单均值将 100 股与 10,000 股等同处理——不合理

Real P&L Impact ▼

Simple mean treats a 100-share order and a 10,000-share order equally — unrealistic. Notional-Weighted IS reflects true P&L impact. (每花出 1块钱，平均损耗多少 bps)

Formula

NW_IS = Σ(IS_i × notional_i) / Σ(notional_i)

Pareto Analysis

Contribution_i = (IS_i × notional_i) / Σ(IS_j × notional_j) × 100%

Sort by notional → top 10% of large orders contribute the majority of total execution cost.

💡 Business Insight: Pareto analysis is extremely useful when facing extreme situations. It shows which orders carry large order size & put high weight on them. This step tells us which algorithm actually spends more (Q2) vs. which algorithm's IS mean is higher (Q1) — these can differ!

Routing decision guidance: Prioritize optimizing high-notional orders.

Q4

特征与 IS 关系验证 · Feature-IS Functional Form

在建模前验证各特征与 IS 的关系形式（线性/非线性），为 Q5 特征变换提供依据

Spearman + LOWESS ▼

Goal: Determine if features are linearly or non-linearly related to IS.

  • •If linear → raw scale → OLS directly
  • •If non-linear → apply log/sqrt transform → then OLS

Two Tools Used

ToolPurposeResult
Spearman ρRank correlation (monotonicity)Captures non-linear monotone relationships
LOWESSNon-parametric smoothingReveals the actual shape of the relationship

Transform Category Logic

  • 1.Multiplication (乘法关系) ΔIS ∝ ΔX → apply log()
  • 2.Square root (方乘关系) ΔIS ∝ √ΔX → apply sqrt() (concave, diminishing returns)
  • 3.Linear (X增加一样值，IS增加是恒值) → no transform

Final Transform Decisions

FeatureRelationshipTransformTheory
volatilityPositive, multiplicativelog(volatility)波动 → 逆向选择成本
averageSpreadPositive, multiplicativelog(averageSpread)价差 → 流动性差
ordersizePositive, square-root√ordersizeAlmgren-Chriss market impact law
durationAmbiguousduration (raw)减少即时冲击 vs 增加时序风险
exec_intensityPositive, square-root√exec_intensity执行速率 → 冲击大
Foundation set: These transforms lay the groundwork for building a theoretically-grounded OLS causal regression in Q5. LOWESS reveals non-linear shape → decides whether to apply log/sqrt.
Q5

核心 OLS 因果回归 · Core OLS Causal Regression

控制订单难度后，algo_2 相比 algo_1 的真实 IS 溢价是多少？

β₇ = −83.58 bps ★ ▼

Theory-driven feature transforms (from Q4) applied. algo_dummy = 1 if algo_2, else 0.

Feature Engineering

log_vol = log(volatility) # 线性化乘法关系 log_spread = log(averageSpread) # 同上 sqrt_size = sqrt(ordersize) # Almgren-Chriss 平方根冲击定律 sqrt_intensity = sqrt(exec_intensity) # POV 代理 spread_x_size = log_spread × sqrt_size # 交互项：双重惩罚 algo_dummy = 1 if algo_2 else 0 # ← 关键哑变量

Full OLS Model

IS = β₀ + β₁·log_vol + β₂·log_spread + β₃·√size + β₄·duration + β₅·√intensity + β₆·spread×size + β₇·algo_dummy + ε

Coefficient Interpretation

CoefficientRolePurpose
β₀InterceptBaseline IS level
β₁–β₆Control variablesExplain & absorb influence of order difficulty (volatility, spread, size, duration, intensity) — 4个因素就是订单难度
β₇ (algo_dummy)★ KEY ANSWERPure algorithm performance — the exact capacity of algorithm, after controlling for all other factors

Decision Rule

ConditionInterpretation
β₇ > 0 and p < 0.05algo_2 is more expensive
β₇ < 0 and p < 0.05algo_2 is cheaper (better)

Q5 Result: β₇ = −83.582, p-value < 0.05

After controlling for order difficulty (volatility, spread, size, duration), algo_2 is cheaper than algo_1 by 83.58 bps. This is statistically significant. Proceed to robustness checks.

Q5R

稳健性检验三层 · Robustness: HC3 + Huber + Quantile

单一 OLS 可能受异方差或残差异常值影响，三层交叉验证

3-Layer Validation ▼

Layer 1: Breusch-Pagan + HC3

测试是否存在异方差现象 — 随着X的增大向直大，还是缩影向

  • H₀: Var(Residual) is constant — Homoskedasticity
  • H₁: Var(Residual) changes with X — Heteroskedasticity
bp_stat, bp_p, _, _ = het_breuschpagan(residuals, X) # p < 0.05 → heteroskedasticity present → must use HC3 standard errors

HC3 (heteroskedasticity-consistent SE) corrects standard errors without changing coefficients.

Layer 2: Huber Regression (ε = 1.35)

Denoise regression — 对大残差降权，让异常值少说话

Error SizeLoss FunctionEffect
|e| < εe² (quadratic)Normal treatment
|e| > ε|e| (linear)Outliers penalized — down-weighted
HuberRegressor(epsilon=1.35, max_iter=500)

Layer 3: Quantile Regression Q10–Q90

OLS fits the mean, but mean is pulled to the right tail by large orders.

Validation goal: Is the cost advantage concentrated in the tail (extreme orders)? (验证目标：成本差异是否集中在尾部（极端难订单）)

for q in [0.10, 0.25, 0.50, 0.75, 0.90]: qr = smf.quantreg(formula, df).fit(q=q, max_iter=2000)

Q6

部分成交与存活偏差 · Partial Fills & Survivorship Bias

题目假设全量成交——Q6 开放此假设，揭示"作弊算法"的存在

⚠ Optional but Critical ▼

The problem assumes full fills. Q6 relaxes this — reveals "cherry-picking algorithms": only execute when market moves favorably, reporting IS ≈ 0, but achieving this via extremely low fill rates.

Expanded IS Formula

Expanded_IS_i = r_i × IS_filled_i + (1 − r_i) × OppCost_i
r_i = filled_qty / order_qty # 成交率 OppCost = IS_Slippage + Normal(50, 80) # 未成交会产生成本 # Cherry-pick algorithm: fill rate negatively correlated with IS cherry_fill = clip(1 - 0.0005 × IS_Slippage + noise, 0.05, 1.0) # Neutral algorithm: fill rate independent of IS neutral_fill = clip(0.6 + noise, 0.05, 1.0)
🚨 Warning: Reporting IS ≈ 0 does NOT mean the algorithm is excellent. Must simultaneously report fill rate + Expanded IS. Otherwise the metric can be gamed. This is a critical audit check.
Q7

LightGBM + SHAP 非线性验证 · Non-Linear Cross-Validation

所有之前的回归都是建立在 linear 之上的 — 用 LightGBM+SHAP 来学习非线性关系，打开黑盒

Final Confirmation ▼

OLS assumes linearity. But market impact at liquidity thresholds can have non-linear jumps. LightGBM captures non-linearity; SHAP provides theoretically rigorous feature attribution.

Model Hyperparameters

n_estimators = 300 # number of trees lr = 0.05 # step size num_leaves = 31 # tree complexity (31 ≈ 2⁵, depth=5) subsample = 0.8 # row sampling early_stop = 30 # stop when no improvement for 30 rounds

→ 直接连续参数很难合出模型 vs LightGBM: 可以无限分段完整结构

关键: OLS 用直线性过效果 vs LightGBM: 可以无限分段完整结构

explainer = shap.TreeExplainer(lgb_model) shap_values = explainer.shap_values(X_test) # Verify algo_dummy direction is consistent with OLS mean_shap_algo2 = shap_values[X_test['algo_dummy']==1, algo_idx].mean()
