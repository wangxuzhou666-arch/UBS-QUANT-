S0.1 Winsorization Outlier Treatment (P1–P99) ▼

交易数据天然有"离群怪物"——就像班级平均身高里混进了姚明，一个极端值就能把整体结论带跑。我们把每列数据掐头去尾，只保留第 1 到第 99 百分位之间的部分（Winsorization），相当于"去掉最高分和最低分再算平均"。后面回归分析用的纠偏工具（HC3 稳健标准误）能处理"大小不均匀"的问题（异方差），但对付不了这种极端怪物——所以必须先动手术再上检测仪。

先做最简单粗暴的比较：直接看两个算法的平均滑点谁高谁低，用三种不同的统计方法交叉印证，防止单一方法说谎：

Welch t-test Mann-Whitney U Bootstrap CI N=10,000 Cohen's d
t_stat, p_val = stats.ttest_ind(g1, g2, equal_var=False) # Welch u_stat, u_p = stats.mannwhitneyu(g1, g2, alternative='two-sided') boots = [np.mean(np.random.choice(g1,n,True)) - np.mean(np.random.choice(g2,n,True)) for _ in range(10_000)]
发现： algo_1 的原始滑点明显低于 algo_2——但这就像说"A 餐厅比 B 餐厅便宜"，你还没看菜单呢。价格差异可能不是厨艺问题，而是 A 只接到了简单菜的订单（路由偏差）。到底是算法真的更强还是"挑了软柿子捏"，要到 Q3 才能验证。

Q2 名义价值加权 IS Notional-Weighted IS + Pareto ▼

简单平均有个致命问题：买 100 股的小单和买 10,000 股的大单被当成同等重要——这就像把买一瓶水和买一辆车的交易放在一起算"平均消费体验"。用名义金额加权（Notional-Weighted IS）才能反映真实盈亏影响——花的钱越多，那笔交易的滑点就越该被重视：

NW_IS = Σ(IS_i × notional_i) / Σ(notional_i)

二八定律验证（Pareto 分析）：把订单按金额从大到小排队，看头部的大额订单到底"吃掉"了多少总成本。

Contribution_i = (IS_i × notional_i) / Σ(IS_j × notional_j) × 100%
实战意义： 这直接告诉交易台一件事——把精力花在优化大单的路由上，回报最大。小单滑点高一点无所谓，大单多滑一个 bps 就是真金白银。
EDA 实证： 加权后滑点（NW_IS）：algo_1 = 206.9 bps，algo_2 = 278.4 bps，差距拉到 +71.5 bps。Pareto 曲线显示前 20% 的大单贡献了约 70% 的累积滑点成本——典型的"二八定律"。algo_2 的累积曲线爬升更陡，说明它在大单上的成本占比更高——大单路由选择是成本控制的核心战场。

Q3 特征分布分析 Feature Distribution + KS Test ▼

想象两个厨师比赛，但裁判偷偷给其中一个更难的食材——这比赛还公平吗？这一步就是检查"裁判有没有作弊"。我们把每个订单特征画成分布曲线（KDE 密度图），再用一个统计裁判（KS 检验）判断两组订单的"难度配方"是否一样。同时，构造一个新指标 exec_intensity（执行强度）= 订单量 ÷ 执行时长，衡量"每秒要消化多少货"——速度越快，噎着的风险越大：

exec_intensity = ordersize / max(duration, 1e-4) # 执行强度越高 → 在更短时间内向市场注入更大量 → 冲击更大 ks_stat, ks_p = stats.ks_2samp(df_algo1[feat], df_algo2[feat]) # p < 0.05 → 特征分布显著不同 → 混淆变量确认
核心发现： 两组订单的"难度配方"显著不同——系统把更大、更急的订单塞给了 algo_2。这就像让一个选手跑 100 米、另一个跑 400 米，然后比谁用时多。Q1 的原始对比结论不可信，必须先"把赛道拉平"（控制变量回归，见 Q5）。
EDA 实证（路由偏差铁证）：
✅ 订单大小 ordersize：algo_2 的 log 中心在 4 附近，algo_1 在 3 附近——algo_2 被分配的订单平均大了一个数量级，相当于别人搬砖你搬石头
✅ 执行强度 exec_intensity：algo_2 整体右移约 1 个单位——不仅订单更大，还被要求"吃得更快"
❌ volatility / averageSpread / duration：两组无差异，说明偏差的根源就是订单大小和执行速度，不是市场环境

Q4 特征与 IS 关系验证 Feature–IS Functional Form ▼

在搭建模型之前，先搞清楚每个因素和交易成本之间到底是"直线关系"还是"弯曲关系"——就像你想知道"多吃一碗饭是不是固定多胖一斤"，还是"前几碗没感觉、第五碗突然胖十斤"。这决定了 Q5 建模时需不需要对数据做"整形手术"（对数/开方变换）：

Spearman ρ LOWESS frac=0.2
特征预期关系原因（微观结构）
volatility正相关 ↑价格像心电图一样剧烈跳动时，对手更容易"看穿你的底牌"趁机占便宜
averageSpread正相关 ↑买一价和卖一价之间的"过路费"越宽，说明这条路越冷清、没人接盘
ordersize正相关 ↑大单入场像大象踩进游泳池——水面不是线性涨，而是越大溅得越猛（√冲击定律）
duration模糊慢慢卖像细水长流不砸价格；但拖太久市场可能风云突变把你吃掉
exec_intensity正相关 ↑单位时间硬塞越多货，像高峰期挤地铁——挤得越急，踩到的脚越多
用途： LOWESS 平滑曲线（一条"不带偏见的手绘趋势线"）揭示了每个特征和成本之间的真实形状，告诉我们 Q5 建模时哪些变量需要先做 log 或 sqrt 变换，让"弯的关系"变直，回归才准。

Q5 核心 OLS 因果回归 Core OLS — Causal Regression ▼

前面发现了"比赛不公平"（Q3）和"因素形状"（Q4），现在终于可以回答核心问题：如果把两个算法放在完全相同难度的订单上，谁更省钱？做法是建一个回归方程，把订单大小、波动率等"难度因素"全部锁死（控制变量），只看算法标签（algo_dummy）这一个开关对成本的净影响。特征按 Q4 发现的形状做变换：

log_vol = log(volatility) # 线性化乘法关系 log_spread = log(averageSpread) # 同上 sqrt_size = sqrt(ordersize) # Almgren-Chriss 平方根冲击定律 sqrt_intensity = sqrt(exec_intensity) # POV 代理 spread_x_size = log_spread × sqrt_size # 交互项：双重惩罚 algo_dummy = 1 if algo_2 else 0 # ← 关键哑变量 IS = β₀ + β₁·log_vol + β₂·log_spread + β₃·sqrt_size + β₄·duration + β₅·sqrt_intensity + β₆·spread_x_size + β₇·algo_dummy + ε
整个分析的最终裁决落在一个数字上——β₇（algo_dummy 的系数）：
β₇ > 0 且 p < 0.05 → algo_2 在同等条件下更贵，每笔多花 β₇ 个基点
β₇ < 0 且 p < 0.05 → algo_2 在同等条件下更省，每笔省 |β₇| 个基点
p ≥ 0.05 → 两者没有真正差别，Q1 看到的差距全是"裁判偏心"造成的幻觉
经典的辛普森悖论（Simpson's Paradox）翻转： β₇ = -83.58 bps（p=0.0000，95% CI: [-86.50, -80.66]）
就像一个医生在重症病房治愈率 90%、在轻症病房治愈率 95%，合并数据后"看起来"不如另一个只看轻症的医生。真相是：
Q1 原始对比：algo_2 贵 +40 bps（"这医生不行"）→ Q5 控制难度后：algo_2 省 -83.6 bps（"这医生其实是全院最强的，只是一直在啃硬骨头"）
方向完全反转，路由偏差被彻底识别和剥离。
技术细节：残差图呈喇叭形（异方差），已用 HC3 修正。系数影响力排序：log_vol 最强（约 +200 bps）> sqrt_size（约 +30 bps）> algo_dummy（-84 bps）。

Q5R 稳健性检验三层 Robustness: HC3 + Huber + Quantile ▼

想象你只用一把尺子量桌子，万一尺子本身弯了呢？我们用三把不同材质的尺子（HC3 白噪声校正、Huber 抗干扰回归、分位数回归）分别量同一张桌子——三把尺子读数一致，才敢说"这张桌子确实是这么长"：

① Breusch-Pagan + HC3 ② Huber ε=1.35 ③ Quantile Q10–Q90
# ① 异方差检验 bp_stat, bp_p, _, _ = het_breuschpagan(residuals, X) # p < 0.05 → 必须用 HC3 标准误 # ② Huber 回归 HuberRegressor(epsilon=1.35, max_iter=500) # ③ 分位数回归 for q in [0.10, 0.25, 0.50, 0.75, 0.90]: qr = smf.quantreg(formula, df).fit(q=q, max_iter=2000)
三重验证逻辑： Huber 回归像一个"自动忽略疯子发言"的投票系统，它和 OLS 对 algo_dummy 的判断方向一致，说明个别极端残差没有带偏结论。分位数回归则逐层切片检查——成本优势是不是只靠几笔极端订单撑场面。
EDA 实证（分位数回归逐层扫描）：
Q10: -63 bps · Q25: -70 bps · Q50: -76 bps · Q75: -80 bps · Q90: -78 bps
好比一个学生不只是某一科考得好，语数外理化生全科都赢——algo_2 的优势从最容易的订单到最难的订单全覆盖，而且订单越难（高分位端），它赢得越多。不是靠一两笔神单撑起来的。

P&L 经济显著性转化 Economic Significance — Dollar Impact ▼

统计学家说"显著"，交易台问的是"显著几块钱"。这一步就是把实验室里的 bps 数字翻译成财务报表上的人民币金额——让决策者一眼看懂值不值得换算法：

Dollar Impact = (β₇ / 10,000) × Total Notional of algo_1 orders # 例：β₇ = +50 bps, Total Notional = 10 亿 CNY # Impact = (50 / 10000) × 1,000,000,000 = 500 万 CNY / year # → algo_2 每年多花 500 万 CNY
翻译官角色： 把学术语言（回归系数 bps）直译为商业语言（年省多少钱），给交易台一个"换还是不换"的明确答案。
EDA 实证： algo_1 全年经手名义金额 = 6,571 亿 CNY。β₇ = -83.58 bps。
换算：(-83.58 / 10,000) × 6,571 亿 = -54.9 亿 CNY/年（节省）
通俗讲：每交易 100 万 CNY，省下 8.36 元执行成本——听起来不多，但乘以六千多亿的体量，一年就是 55 亿。这笔钱不需要任何额外风险，纯粹是"换条路走，少交过路费"。
建议：立即启动算法迁移评估。
Dollar impact 确立 → 警惕存活偏差作假
Q6 部分成交与存活偏差 Partial Fills & Survivorship Bias ▼

市场里有一种"作弊算法"：它像一个只在顺风时才出门跑步的人，然后宣称自己从不淋雨。具体手法——只在价格朝有利方向动的时候才成交，成交不了的单悄悄丢掉。报告出来的 IS 接近零，看起来完美，但成交率极低，真实成本被藏起来了。Q6 专门拆穿这类把戏。

Expanded_IS_i = r_i × IS_filled_i + (1 - r_i) × OppCost_i r_i = filled_qty / order_qty # 成交率 OppCost = IS_Slippage + Normal(50, 80) # 未成交机会成本 # 作弊算法：成交率与 IS 负相关 cherry_fill = clip(1 - 0.0005 × IS_Slippage + noise, 0.05, 1.0) # 中性算法：成交率独立于 IS neutral_fill = clip(0.6 + noise, 0.05, 1.0)
核心警示： 报告 IS ≈ 0 不等于算法优秀，就像"考试只交自己会做的题然后宣称正确率 100%"。必须同时公示成交率 + Expanded IS（把未成交单的机会成本算回去），否则指标可被蓄意美化。
EDA 实证： 模拟的 cherry-picker 算法成交率几乎全部堆在 1.0 附近——它精准回避了那些高成本的难单，只吃"白送的肉"。
Filled-only IS（有偏）= 223 bps · Cherry Expanded = 224 bps · Neutral Expanded = 233 bps
偏差仅 10 bps（约 4.3%）。好消息：Q5 的主结论（83.6 bps 优势）完全经得起这层检验。但方向很清楚——cherry-picking 确实会让算法的成绩单比实际好看一点。

Q7 LightGBM + SHAP 非线性验证 Non-Linear Cross-Validation ▼

前面所有分析都假设"原因和结果之间是一条直线"。但真实市场更像山路——流动性突然枯竭时，冲击成本不是缓慢上升，而是断崖式跳涨。LightGBM 是一个能自动识别弯道和断崖的模型，SHAP 则像 X 光片，让我们看清每个因素在模型内部到底贡献了多少：

n_estimators=300 lr=0.05 num_leaves=31 subsample=0.8 early_stop=30
explainer = shap.TreeExplainer(lgb_model) shap_values = explainer.shap_values(X_test) # 验证 algo_dummy 方向是否与 OLS 一致 mean_shap_algo2 = shap_values[X_test['algo_dummy']==1, algo_idx].mean()
Shapley 公理含义
效率性所有 SHAP 值之和 = 预测值 − 基准值
对称性贡献相同的特征获得相同 SHAP
虚拟性无影响特征 SHAP = 0
可加性组合模型 SHAP = 各模型之和
终极交叉验证： 线性模型（OLS）说 algo_2 更好，非线性模型（LightGBM + SHAP）也说 algo_2 更好——两个世界观完全不同的裁判，判决一致。结论与方法论无关，是事实本身的属性。
EDA 实证（SHAP 特征重要性排名）：
sqrt_size ≈ 75 · log_vol ≈ 60 · algo_dummy ≈ 30 · sqrt_intensity ≈ 12 · log_spread ≈ 2
翻译：订单大小和波动率是影响执行成本的两座大山，算法选择排第三——虽然不是最大因素，但它是唯一一个交易台可以主动选择的因素。
algo_dummy 的 SHAP 值：当 algo_2=1 时，点几乎全部落在负值区（降低成本）→ 与 OLS β₇ < 0 完全吻合。
log_spread SHAP 几乎贴着零线 → 买卖价差对执行成本的贡献微乎其微，与 Spearman ρ=0.002 严丝合缝。
